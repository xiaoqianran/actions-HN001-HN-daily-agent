# 摘要提示词模板（可选）

可将模板复制到 GitHub Actions Variable `SUMMARY_PROMPT_TEMPLATE`。  
**必须包含** `{title}` 与 `{content}`。

> 推送侧会把模型输出清洗为「核心 / 要点 / 亮点 / 适合」四段，并用 HTML 排版。  
> 自定义提示词时，建议仍输出这四个标签，排版最稳。

---

## 模板 0：默认结构化（推荐，与代码内默认一致）

```text
你是技术日报编辑。请根据文章生成中文摘要，供微信推送阅读。

标题：{title}
正文：
{content}

【输出要求——必须严格遵守】
1. 只输出下面 4 行结构，不要前言、不要后记、不要代码块、不要标题符号（#）、不要表情符号。
2. 每行以固定标签开头，标签后紧跟一个空格，再写正文。
3. 「核心」一行，不超过 40 个汉字。
4. 「要点」恰好 3 条，用中文分号「；」分隔，每条不超过 25 个汉字。
5. 「亮点」一行，一句最有价值的信息，不超过 30 个汉字；没有则写「无」。
6. 「适合」一行，标注读者画像（如：后端/前端/AI/产品/安全），不超过 15 个汉字。
7. 禁止输出思考过程、英文长句、Markdown 列表（- * 1.）、加粗符号 **。

【标准输出格式示例】
核心 这是一款用 Rust 重写的 JS 运行时，显著降低冷启动延迟。
要点 用 Rust 重写核心循环；兼容大部分 Node API；基准测试延迟下降约 40%
亮点 已有生产环境案例，可渐进迁移。
适合 后端与基础设施工程师

现在按上述格式输出，不要输出示例本身。
```

## 模板 1：极简快讯

```text
请把以下内容压成四行中文摘要。
标题：{title}
正文：{content}

只输出：
核心 <一句话，≤40字>
要点 <三点，用；分隔>
亮点 <一句或无>
适合 <读者画像>
禁止其它任何文字。
```

## 模板 2：工程实践

```text
面向工程师总结以下文章。
标题：{title}
正文：{content}

只输出四行：
核心 <解决了什么问题>
要点 <技术点1；技术点2；技术点3>
亮点 <可落地的一条建议>
适合 <角色，如后端/SRE>
不要 Markdown，不要列表符号。
```
